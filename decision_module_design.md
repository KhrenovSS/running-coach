# Архитектура модуля аналитики и рекомендаций — AI Running Coach

> Документ описывает архитектуру модуля, который после каждой загруженной тренировки
> анализирует её в контексте истории, метрик восстановления и литературы, а затем
> рекомендует **какую** следующую тренировку провести и **когда**.
>
> Документ рассчитан на разработчика любого уровня: каждый этап содержит «что это»,
> «зачем», «какие файлы создать» и «как проверить».

---

## 0. Что уже есть в проекте (отправная точка)

Чтобы не проектировать в вакууме — фиксируем текущее состояние (см. `AGENTS.md`, `README.md`, `src/models.py`):

- **Стек:** Python + FastAPI + SQLite (SQLAlchemy) + `python-telegram-bot` + APScheduler.
- **Готово:** парсинг TCX/FIT, классификация тренировок (интервальная/темповая/long/recovery),
  сегментация, пульсовые зоны, погода, синхронизация Coros, загрузка метрик здоровья.
- **Модели данных (`src/models.py`):**
  - `User` — профиль, цель (`goal_type`), уровень (`sport_level`), `max_hr`, `lthr` (косвенно).
  - `TrainingSession` — тренировка: дистанция, HR, тип, сегменты, `training_effect`, `vo2max`, `calories`.
  - `DailyMetrics` — ежедневные метрики: `avg_sleep_hrv`, `sleep_hrv_baseline`, `rhr`,
    `tired_rate`, `training_load`, `ati`, `cti`, `performance`, `vo2max`, `lthr`, `stamina_level`, `ltsp`.
  - `WeightMeasurement` — вес.
  - **`TrainingFeedback` — оценка тяжести тренировки 0–10 + заметка.** ← ключевой кирпич для обучения.
- **База знаний по метрикам:** `docs/coros_health_metrics.md` — численные пороги и 8 IF-THEN правил.
  Это, по сути, **уже готовый фундамент детерминированного движка** — его надо лишь перенести в код.

**Чего ещё нет:** папки с книгами по бегу, движка решений, слоя персонализации/обучения,
LLM-интерфейса, новых таблиц (рекомендации, прогнозы, «уроки», персональная модель).

## 1. Что было в прошлой версии этого документа и что меняем

Прошлый дизайн (слоистая архитектура, «LLM как интерфейс, а не движок», детерминированные
правила безопасности, RAG для книг) — **в целом верный и мы его сохраняем**. Но два пункта,
которые критичны по ТЗ, были проработаны слабо. Исправляем:

| Проблема прошлого дизайна | Решение в этой версии |
|---|---|
| RAG-литература стояла как «правило приоритета 3» и предполагала анализ книги в рантайме. | Литература **дистиллируется ОФЛАЙН** в тематические файлы-руководства (`guides/`) с машиночитаемыми правилами. Книга анализируется **один раз**, дальше движок читает компактные руководства. RAG остаётся только для свободного чата (цитаты). |
| «Адаптивность» упоминалась лозунгом, без механизма. | Введён отдельный **слой персонализации и обучения**: цикл «прогноз → факт → расхождение → разбор причины → корректировка персональной модели». «Неожиданно тяжёлые» тренировки детектируются автоматически и разбираются. |
| Не было ответа на вопрос ТЗ «**когда** проводить тренировку». | Введён модуль **Recovery Timing** (правило приоритета 2): движок считает время до восстановления и предлагает дату/окно. |
| Не хватало схемы данных для решений. | Добавлены таблицы `Recommendation`, `PredictionLog`, `UserModel`, `Lesson`. |

## 2. Главные принципы архитектуры

1. **LLM = интерфейс, а не источник решений.** Все числовые решения (тип, темп/зона, дата)
   принимает детерминированный движок. LLM только объясняет и общается. Это безопасность + объяснимость.
2. **Детерминированное ядро = приоритетные правила.** Сначала безопасность, потом всё остальное.
   Каждое решение сопровождается трассировкой («почему так»).
3. **Знания из книг готовятся заранее.** Книга → один раз через пайплайн дистилляции →
   тематические руководства (`guides/*.md` с YAML-параметрами). Движок и LLM читают руководства, не книги.
4. **Система учится, но предсказуемо.** Обучение = калибровка персональных параметров на основе
   расхождения «прогноз/факт» и подтверждённых пользователем причин. Никакого чёрного ящика над безопасностью.
5. **Всё детерминированное — тестируемое.** Каждый аналитический «скилл» и каждое правило
   покрываются юнит-тестами на фикстурах (заранее заготовленных данных).
6. **Кэширование и идемпотентность.** Тяжёлые расчёты кэшируются; повторная обработка одной
   тренировки не ломает данные.

## 4. Слой Analytics / Skills (детерминированная аналитика)

«Скилл» — это **чистая функция**: получает данные из БД, возвращает структурированный результат.
Никаких побочных эффектов, легко тестировать. Единый формат результата:

```python
@dataclass
class SkillResult:
    key: str                 # "hrv", "ati_cti", "running_efficiency", ...
    value: float | None      # числовое значение, если есть
    status: str              # "ok" / "warning" / "danger" / "unknown"
    confidence: float        # 0..1 — насколько доверяем (мало данных → ниже)
    message: str             # человекочитаемое объяснение
    evidence: dict           # сырые числа, на которых основан вывод (для трассировки)
```

Какие скиллы и что считают (формулы и пороги уже есть в `docs/coros_health_metrics.md`):

| Скилл | Что делает | Откуда формулы |
|---|---|---|
| `fatigue.py` | HRV-статус (Повышенная/Норма/Пониженная/Низкая), RHR-аномалия, классификация `tired_rate` | разделы 2,3,6 |
| `load.py` | ATI/CTI ratio, Training Status (Base Fitness/Load Impact/Intensity Trend), недельный TL | разделы 7,11,19 |
| `distribution.py` | доля времени/нагрузки по зонам за 7/28 дней, проверка баланса 80/20 | раздел 18 + книги |
| `progress.py` | тренды (slope) VO2max, LTHR, stamina за 30/90 дней | раздел 9,10 |
| `recovery.py` | Recovery % (маппинг `performance`), **оценка часов до восстановления** по типу нагрузки | разделы 12,21,22 |
| `workout.py` | разбор завершённой тренировки: тип, Training Effect, Running Efficiency, попадание в целевые зоны | разделы 13,14,15 |

**Важно:** `confidence` снижается, когда данных мало (новый пользователь, нет HRV-нормы за 30 дней).
Движок учитывает доверие: при низком доверии — консервативнее.

**Как проверить:** для каждого скилла — `tests/skills/test_<skill>.py` с фикстурами
(набор `DailyMetrics`/`TrainingSession`), где вручную посчитан ожидаемый результат.

## 5. State Assessor — единая картина состояния

`state.py` собирает результаты всех скиллов в один снимок `AthleteState`:

```python
@dataclass
class AthleteState:
    user_id: int
    as_of: datetime
    readiness_score: float          # 0..100 — готовность к интенсиву
    fatigue_score: float            # 0..100 — накопленная усталость
    injury_risk: float              # 0..1
    recovery_hours_left: float      # сколько ещё восстанавливаться
    hrv_status: str                 # normal / low / very_low / high
    ati_cti_ratio: float | None
    zone_balance: dict              # {"z1_z2": 0.78, "z3_plus": 0.22}
    last_workout: dict | None       # тип, нагрузка, эффект последней тренировки
    progress: dict                  # тренды
    skills: list[SkillResult]       # всё сырьё для трассировки и LLM
    data_confidence: float          # общее доверие к данным
```

`readiness_score` и `fatigue_score` — интегральные числа, собранные из скиллов по прозрачной
взвешенной формуле (веса вынести в конфиг, чтобы потом калибровать). Снимок кэшируется
(например, на 1–6 часов) и пересчитывается при новой синхронизации.

## 6. Knowledge Base — литература как ФУНДАМЕНТ (ключевая часть ТЗ)

Цель: **не анализировать книгу каждый раз**. Анализируем один раз офлайн и сохраняем готовые
тематические руководства.

### 6.1. Офлайн-пайплайн дистилляции (`knowledge/distill.py`)

Запускается **вручную** разработчиком при добавлении новой книги (это не часть рантайма):

1. Положить книгу в `knowledge/sources/` (PDF/EPUB).
2. Извлечь текст, нарезать на крупные тематические секции.
3. С помощью LLM (разово, в пакетном режиме) свести каждую тему к структурированному
   **руководству** и сохранить в `knowledge/guides/<topic>.md`.
4. Разработчик **просматривает и правит** руководства (контроль качества) и коммитит их в git.

Темы руководств (минимальный набор):

```
guides/
├── easy_runs.md              # лёгкий бег: зачем, темп = LT pace + 60–90 с/км
├── long_runs.md              # длительные: объём, темп, частота
├── threshold.md              # пороговые: структура, длительность, ЧСС Z4
├── intervals_vo2max.md       # VO2max-интервалы: формат 3–5 мин, отдых, частота
├── recovery_principles.md    # принципы восстановления, разгрузка
├── load_progression_80_20.md # рост нагрузки ≤10%/нед, цикл 3:1, правило 80/20
├── tapering.md               # подводка к старту
├── injury_prevention.md      # профилактика травм, флаги риска
└── plan_templates_10k.md     # шаблоны планов по целям (10k/half/marathon)
└── ...
```

### 6.2. Формат руководства (двойного назначения)

Каждый файл = **YAML-заголовок с машиночитаемыми правилами** + **проза для LLM**:

```markdown
---
topic: easy_runs
sources: ["Daniels' Running Formula, ch.4", "80/20 Running, ch.2"]
key_rules:
  easy_pace_offset_sec_per_km: [60, 90]   # медленнее порогового темпа
  easy_hr_zone: ["Z1", "Z2"]
  max_share_of_weekly_volume: 0.80         # правило 80/20
  weekly_volume_increase_max_pct: 10
citations:
  - "Лёгкий бег должен быть действительно лёгким — разговорный темп."
---

## Зачем нужен лёгкий бег
(проза для LLM: объяснения, нюансы, цитаты — подаётся в промпт при общении)
```

- **`key_rules`** напрямую читает детерминированный движок (правило P3/P4) — это «знания из книг в числах».
- **Проза** идёт в LLM для объяснений и в RAG для свободных вопросов в чате.

### 6.3. Загрузка и поиск (`loader.py`, `rag.py`)

- `loader.py` при старте читает все `guides/*.md`, парсит YAML → отдаёт движку словарь правил.
- `rag.py` строит эмбеддинги **по руководствам** (а не по сырым книгам) для чата: на вопрос
  пользователя находит релевантные фрагменты руководств и подаёт их в LLM.
- Векторная БД: на старте достаточно `sqlite` + `sentence-transformers` (локально, бесплатно);
  при росте — Chroma/Qdrant.

### 6.4. Версионирование знаний

Добавили книгу → запустили `distill.py` только для неё → проверили новые/изменённые руководства →
коммит. Старые руководства не перегенерируются без необходимости.

## 7. Слой Personalization & Learning — адаптивность (ключевая часть ТЗ)

Задача: система должна **учиться** и учитывать, что «эта тренировка оказалась неожиданно тяжёлой»,
понимать **почему** и корректировать поведение. Делаем это прозрачно и безопасно.

### 7.1. Персональная модель (`UserModel`)

Хранит откалиброванные под конкретного человека параметры:

```python
# таблица user_model (1 строка на пользователя), json + типизированные поля
{
  "hrv_baseline": 48.0, "hrv_sd": 9.0, "rhr_baseline": 52,
  "pace_hr_efficiency": {...},        # персональная кривая «темп ↔ пульс»
  "load_tolerance": 1.0,              # множитель переносимости нагрузки (калибруется)
  "recovery_hours_by_type": {         # сколько ЧАСОВ восстанавливается после типа
      "intervals": 48, "threshold": 36, "long": 30, "tempo": 24, "recovery": 12
  },
  "preferences": {"days": ["пн","ср","пт","сб"], "dislikes": ["дорожка"]},
  "injuries": ["левое колено"],
  "updated_at": "..."
}
```

Изначально заполняется дефолтами по уровню (`sport_level`) из руководств, дальше — калибруется.

### 7.2. Прогноз перед рекомендацией (`predictor.py`)

Когда движок предлагает тренировку, predictor оценивает **ожидаемые** показатели:

- ожидаемый средний пульс при целевом темпе (по персональной кривой `pace_hr_efficiency`);
- ожидаемая субъективная тяжесть (predicted RPE, шкала 0–10);
- ожидаемая `training_load`.

Эти прогнозы сохраняются вместе с рекомендацией.

### 7.3. Цикл калибровки (`calibration.py`) — сердце «обучения»

После загрузки выполненной тренировки:

```
1. Берём прогноз (predicted) из рекомендации, которой соответствует тренировка.
2. Считаем факт (actual): реальный avg HR, фактический training_load,
   subjective rating из TrainingFeedback (0–10 = тяжесть).
3. residual = actual − predicted   (по каждому показателю)
4. Пишем строку в PredictionLog.
5. Плавно (EWMA, с ограничителями) обновляем UserModel:
   - кривую pace↔HR (стал эффективнее/менее эффективен);
   - load_tolerance;
   - recovery_hours_by_type.
```

Обновления **ограничены сверху/снизу** (нельзя резко менять параметры из-за одной тренировки) —
это защита от выбросов.

### 7.4. Детект «неожиданно тяжело» и разбор причины (`lessons.py`)

```
ЕСЛИ |residual по тяжести| или |residual по HR| превышает порог
     (тренировка оказалась заметно тяжелее прогноза):
  1. Собираем контекст за этот день/ночь:
     HRV той ночи, RHR, tired_rate, жара (avg_temperature), набор высоты,
     дней с последней тяжёлой, изменение веса, время суток, сон.
  2. Эвристика причин + (опционально) LLM формирует ГИПОТЕЗУ:
     "жара 28°C", "HRV ниже нормы", "мало восстановления (36ч < 48ч норма)",
     "слишком быстрый целевой темп", "возможна болезнь/недосып".
  3. Спрашиваем пользователя коротким опросом в Telegram:
     «Тренировка была тяжелее обычного. Что повлияло?
      [Плохой сон] [Стресс] [Жара] [Заболеваю] [Просто тяжёлый день] [Другое]»
  4. Сохраняем Lesson:
     {trigger: {...условия...}, cause: "жара>27", adjustment: {...}, source: "user"}
```

### 7.5. Применение «уроков» (обратно в решения)

Сохранённые `Lesson` влияют на будущие рекомендации через правило **P5**:

- Повторяется условие урока (например, прогноз жары > 27°C) → движок заранее смягчает
  целевой темп / переносит тяжёлую тренировку.
- Накапливается несколько «тяжело при низкой HRV» → снижаем `load_tolerance` или поднимаем
  порог готовности для интенсива.

Так система **корректирует своё поведение**, оставаясь объяснимой: к каждой корректировке
привязан конкретный урок и его причина.

> Примечание о ML: на старте — простые интерпретируемые правила и EWMA. Когда накопится
> достаточно `PredictionLog`, можно добавить регрессор для прогноза тяжести — **но строго под**
> правилами безопасности P1 (ML только уточняет прогноз, не отменяет запреты).

## 8. Decision Engine — приоритетные правила (что и когда)

Движок прогоняет `AthleteState` + `UserModel` + `key_rules` из руководств через **стек правил
по приоритету**. Каждое правило получает черновой `Prescription`, может его изменить/ужесточить
и **обязано записать причину** в трассировку.

```python
@dataclass
class Prescription:
    when: dict                 # {"date": ..., "window": "утро/вечер", "earliest": ...}
    workout_type: str          # recovery / easy / long / tempo / threshold / intervals / rest
    target: dict               # {"hr_zone": "Z2"} или {"pace_min_km": 5.3} или структура интервалов
    volume: dict               # {"distance_km": 8} или {"duration_min": 50}
    rationale: list[str]       # трассировка: почему именно так (по каждому правилу)
    confidence: float
    alternatives: list[dict]   # запасные варианты
    predicted: dict            # прогноз тяжести/HR/нагрузки (из predictor)
```

| Приоритет | Правило | Что делает | Может ли запретить? |
|---|---|---|---|
| **P1** | `p1_safety` | здоровье: HRV low/very_low → только Z1–Z2 или отдых; RHR +10 → отдых; ATI/CTI>1.5 → разгрузка | **Да** (жёстко ограничивает) |
| **P2** | `p2_recovery_timing` | считает `recovery_hours_left` по типу прошлой нагрузки → задаёт **дату/окно** следующей; если рано — даунгрейд или перенос | Сдвигает дату |
| **P3** | `p3_training_logic` | последовательность (после VO2max нельзя threshold), баланс 80/20, цикл 3:1 — из руководств | Меняет тип |
| **P4** | `p4_plan_goal` | подгоняет под план цели (10k/half/marathon) из `plan_templates_*` | Уточняет |
| **P5** | `p5_personal` | предпочтения (дни/нелюбимое), травмы, **уроки** из слоя обучения | Корректирует |

Правила работают **каскадом**: P1 задаёт жёсткие границы (потолок интенсивности/объёма),
нижние приоритеты выбирают конкретику внутри этих границ. Это гарантирует, что персонализация
или план никогда не «пробьют» безопасность.

`prescriber.py` собирает итог, добавляет `predicted` от predictor’а и пишет в таблицу `Recommendation`.

## 9. LLM Coach — интерфейс общения

`llm/coach.py` берёт **готовый** `Prescription` + `AthleteState` + релевантную прозу из
руководств и формирует естественный ответ для Telegram. LLM **не меняет числа** — только
переводит их в человеческий язык, объясняет «почему» и отвечает на вопросы.

```
Промпт (низкая температура 0.1–0.3):
  Роль: дружелюбный, но требовательный тренер по бегу.
  Состояние спортсмена: <AthleteState, кратко>
  Рекомендация движка (НЕ менять числа): <Prescription>
  Справка из руководств: <фрагменты guides через RAG>
  Запрос пользователя: <текст>
  Задача: объяснить рекомендацию простым языком, ответить на вопрос,
          при необходимости — мотивировать/предупредить.
```

Сценарии:
- **Свободный вопрос** («почему сегодня нельзя интервалы?») → RAG по руководствам + State → ответ.
- **Объяснение рекомендации** → пересказ `rationale`.
- **Сбор обратной связи** → запись `TrainingFeedback` и запуск калибровки.

## 10. Orchestrator и триггеры

`orchestrator.py` связывает всё и вызывается из бота/планировщика:

1. **`on_workout_completed(user, session)`** — после загрузки/синхронизации тренировки:
   ```
   workout.py разбирает тренировку
   → calibration.py: прогноз vs факт, обновление UserModel, запись PredictionLog
   → lessons.py: если «неожиданно тяжело» → гипотеза причины + опрос пользователя
   → state.py: пересчёт AthleteState
   → engine.py: новая рекомендация (что + когда)
   → Telegram: сообщение пользователю (через LLM Coach) + кнопки оценки тяжести
   ```
2. **`morning_check(user)`** — утренний джоб (уже есть проверка восстановления в 10:00):
   пересчитать готовность по свежей ночной HRV → подтвердить/скорректировать план на день.
3. **`handle_chat(user, message)`** — входящее сообщение в Telegram → LLM Coach.

## 11. Новые таблицы БД (добавить в `src/models.py`)

```python
class Recommendation(Base):        # что система порекомендовала
    id, user_id, created_at, for_date
    workout_type, target_json, volume_json
    rationale_json                 # трассировка причин
    predicted_json                 # прогноз тяжести/HR/нагрузки
    confidence
    status                         # proposed / accepted / done / skipped
    linked_session_id              # связь с выполненной тренировкой (для калибровки)

class PredictionLog(Base):         # прогноз vs факт (для обучения)
    id, user_id, session_id, created_at
    predicted_json, actual_json
    residual_effort, residual_hr, residual_load
    flagged_hard                   # bool: «неожиданно тяжело»

class UserModel(Base):             # персональные откалиброванные параметры
    id, user_id (unique), params_json, updated_at

class Lesson(Base):                # извлечённый «урок»
    id, user_id, created_at
    trigger_json                   # при каких условиях
    cause                          # подтверждённая/гипотетическая причина
    adjustment_json                # как корректировать поведение
    source                         # "user" / "auto"
    active                         # bool
```

## 12. Поэтапный план разработки

Принцип: **на каждом этапе — работающий результат**, который можно проверить в Telegram.
Этапы идут от безопасного простого к сложному. Двигаемся строго по порядку.

### Этап 0. Каркас и данные (фундамент)
**Зачем:** подготовить структуру, чтобы дальше только наполнять.
- [ ] Создать пакет `src/coach/` со скелетами файлов из раздела 3.
- [ ] Добавить таблицы `Recommendation`, `PredictionLog`, `UserModel`, `Lesson` (+ автомиграция).
- [ ] Завести `src/coach/config.py` (веса формул, пороги, лимиты калибровки — всё в одном месте).
- [ ] Настроить папку `tests/` и фикстуры (пример пользователя с историей `DailyMetrics`/тренировок).
- **Проверка:** `init_db()` создаёт таблицы; тесты-заглушки запускаются.

### Этап 1. Аналитика (Skills) + State Assessor
**Зачем:** превратить пороги из `coros_health_metrics.md` в код.
- [ ] Реализовать `skills/fatigue.py`, `load.py`, `recovery.py` (минимальный набор).
- [ ] Реализовать `skills/distribution.py`, `progress.py`, `workout.py`.
- [ ] Реализовать `state.py` → `AthleteState` (readiness/fatigue/injury_risk/recovery_hours_left).
- [ ] Юнит-тесты на фикстурах для каждого скилла.
- **Проверка:** команда `/state` в боте выводит сводку состояния (без рекомендаций).

### Этап 2. Базовый движок + правила безопасности (P1) + Recovery Timing (P2)
**Зачем:** первые реальные рекомендации «что и когда», безопасные.
- [ ] `rules/base.py` (интерфейс Rule, Prescription, трассировка).
- [ ] `rules/p1_safety.py` — перенести 8 правил из `docs/coros_health_metrics.md`.
- [ ] `rules/p2_recovery_timing.py` — оценка времени до восстановления → дата/окно.
- [ ] `prescriber.py` + запись в `Recommendation`.
- [ ] Команда `/recommend` в боте: тип + когда + причина.
- **Проверка:** при низкой HRV бот рекомендует отдых/Z2 и объясняет; при свежем состоянии — допускает интенсив.

### Этап 3. База знаний из литературы (дистилляция в руководства)
**Зачем:** литература как фундамент, без анализа книги в рантайме.
- [ ] Завести `knowledge/sources/` (книги: Daniels, 80/20 Running — для начала) и `.gitignore` на них.
- [ ] Написать `knowledge/distill.py` (офлайн): книга → черновые `guides/*.md` с YAML.
- [ ] Вручную выверить 4–6 базовых руководств (easy/long/threshold/intervals/load_progression).
- [ ] `knowledge/loader.py` — загрузка `key_rules` в движок.
- [ ] `rules/p3_training_logic.py` и `rules/p4_plan_goal.py` — используют `key_rules`.
- **Проверка:** рекомендации учитывают 80/20 и правильную последовательность типов; в причинах есть ссылка на руководство.

### Этап 4. Персонализация и обучение
**Зачем:** учёт «неожиданно тяжёлых» тренировок и адаптация.
- [ ] `personalization/user_model.py` (+ инициализация дефолтами по уровню).
- [ ] `personalization/predictor.py` — прогноз HR/тяжести/нагрузки.
- [ ] `personalization/calibration.py` — residual, обновление UserModel, `PredictionLog`.
- [ ] `personalization/lessons.py` — детект «тяжело», гипотеза причины, опрос в Telegram, запись `Lesson`.
- [ ] `rules/p5_personal.py` — применение предпочтений, травм и «уроков».
- [ ] Встроить цикл в `orchestrator.on_workout_completed()`.
- **Проверка:** после «тяжёлой» тренировки бот спрашивает причину; в `PredictionLog`/`Lesson` появляются записи; следующая рекомендация учитывает урок.

### Этап 5. LLM Coach (интерфейс + чат)
**Зачем:** живое общение и объяснения.
- [ ] `llm/client.py` (обёртка над API, ключ в `.env`, низкая температура).
- [ ] `llm/coach.py` — промпт из Prescription+State+руководства.
- [ ] `knowledge/rag.py` — эмбеддинги по руководствам для свободных вопросов.
- [ ] Подключить к Telegram: свободные вопросы + объяснение рекомендаций.
- **Проверка:** на вопрос «почему сегодня лёгкий бег?» бот объясняет человеческим языком, опираясь на состояние и руководства; числа совпадают с движком.

### Этап 6. Планировщик многонедельных планов
**Зачем:** не только «следующая тренировка», но и план под цель.
- [ ] `plan_templates_*` руководства (10k/half/marathon) → генератор недельного/блочного плана.
- [ ] Учёт цикла 3:1, подводки (tapering), текущего состояния.
- [ ] Команды `/plan`, пересчёт плана при отклонениях.
- **Проверка:** бот строит план на 4–12 недель и адаптирует его, когда состояние/факт расходятся с планом.

### Этап 7. Обратная связь, мониторинг и оценка качества
**Зачем:** убедиться, что система улучшается, и настраивать её.
- [ ] Логировать все рекомендации с причинами (уже частично в `Recommendation`).
- [ ] Метрики качества: доля принятых рекомендаций, средняя оценка тяжести vs прогноз, точность прогноза HR.
- [ ] Возможность переключать наборы правил/веса (для сравнения, A/B).
- [ ] Периодический отчёт прогресса в Telegram (похвала/предупреждение).
- **Проверка:** дашборд/команда `/quality` показывает, насколько точны прогнозы и приняты ли рекомендации.

## 13. Технические заметки

- **LLM:** внешний API (OpenAI/Anthropic) или локальная модель; температура 0.1–0.3; ключи в `.env`.
- **Эмбеддинги/RAG:** на старте — `sentence-transformers` локально + хранение в SQLite; при росте — Chroma/Qdrant.
- **Кэш состояния:** `AthleteState` кэшировать на 1–6 часов; инвалидация при новой синхронизации.
- **Идемпотентность:** повторная обработка одной тренировки не должна задваивать калибровку
  (защита по `session_id` в `PredictionLog`).
- **Безопасность калибровки:** все обновления `UserModel` — с жёсткими min/max и сглаживанием (EWMA).
- **Тесты:** детерминированные части (скиллы, правила, калибровка) покрываем юнит-тестами на фикстурах;
  LLM-часть — отдельно, с моками.
- **Стиль:** комментарии на русском с английским в скобках, changelog и коммиты — по правилам `AGENTS.md`.
